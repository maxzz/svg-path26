**Decisions**
- The generator selector will live in the Export SVG dialog only, not in the global Options dialog.
- Option 1 and option 4 will be implemented in separate files, with a shared helper layer for normalization/source building.
- Option 4 will use `@svgr/core` plus `@svgr/plugin-jsx`, but only if a first-pass browser-runtime spike proves it can bundle and execute entirely in the client; existing `svgo` handling remains outside SVGR to avoid double-optimizing.
- The exported filename must be lowercase even if the generated React component name remains PascalCase inside the file.
- The initial implementation scope keeps the existing SVG code accordion; it does not require a full React-code preview UI unless implementation proves that generator debugging needs one.

**Further Considerations**
1. `@svgr/core` is documented primarily as a Node/tooling API, and the official SVGR playground performs browser-side requests to a remote transform API rather than running the full transform locally. Because this app must run entirely in the browser, the first implementation pass should treat local browser-runtime compatibility as a concrete validation checkpoint. If Vite bundling or runtime execution fails, stop and re-decide rather than silently substituting a remote service or build-time-only path.
2. Tailwind conversion should stay on a safe whitelist first; any attribute that does not map cleanly should remain a JSX prop so option 1 and option 4 both preserve semantics.
3. If `@svgr/core` is not browser-viable, keep the two-generator UI but replace option 4 with a second fully local generator: a markup-driven converter that parses the normalized SVG with `DOMParser` or the existing `SvgInputDocument` model, rewrites attributes to JSX/Tailwind, and emits TSX without any server calls. This is the preferred fallback because it preserves the browser-only constraint and still gives users two distinct generation strategies.
4. A heavier fallback exists if stronger transformation guarantees are needed: use a browser-bundled AST/codegen stack such as `svg-parser` plus Babel/code generator packages. That remains client-side, but it is materially larger and slower than the DOM/model-driven fallback, so it should be treated as a last resort.
